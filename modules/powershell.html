01

Les bases du langage

PowerShell n'est pas un shell de texte mais un shell orienté objet : chaque commande renvoie de vrais objets .NET que l'on peut filtrer, trier et chaîner dans un pipeline.

Le verbe-nom : une grammaire prévisible

Toutes les commandes (cmdlets) suivent le motif Verbe-Nom : Get-, New-, Set-, Remove-, Install-… Connaître le verbe suffit souvent à deviner la commande.

PowerShell — découverte
# Chercher une commande
Get-Command -Verb New -Noun *VM*

# Lire l'aide et des exemples
Get-Help New-VM -Examples

# Inspecter les propriétés d'un objet
Get-NetAdapter | Get-Member

# Pipeline : filtrer puis afficher
Get-Service | Where-Object Status -eq "Running" | Select-Object Name, Status

Réflexe

Bloqué ? Get-Help <cmdlet> -Examples et <objet> | Get-Member répondent à 90 % des questions sans quitter la console.

02

Créer des VM Hyper-V

Toute la mise en place du labo commence par un commutateur virtuel privé et des VM de génération 2 créées sur des disques différentiels pointant vers l'image SYSPREP de référence.

Le commutateur et un serveur

PowerShell — hôte Hyper-V
# Un réseau privé isolé pour le labo
New-VMSwitch -Name "LABOPS" -SwitchType Private

# Disque enfant basé sur l'image généralisée
New-VHD -Path .\SRV-AD01-DISK1-DIFF.vhdx `
    -ParentPath "C:\PARENT\TEST-SYSPREP.vhdx" -Differencing

# La VM (Gen 2, 8 Go, branchée sur LABOPS)
New-VM -Name "SRV-AD01" -Generation 2 -MemoryStartupBytes 8GB `
    -VHDPath .\SRV-AD01-DISK1-DIFF.vhdx -Path .\ -SwitchName "LABOPS"

# Mémoire statique + 2 vCPU
Set-VMMemory -VMName SRV-AD01 -DynamicMemoryEnabled $false
Set-VMProcessor -VMName SRV-AD01 -Count 2

Un client Windows 11 (ISO + TPM)

Un client part d'un disque vierge, démarre sur l'ISO, et exige le TPM virtuel + Secure Boot requis par Windows 11.

PowerShell — VM client
New-VHD -Path .\CLIENT-WIN11-DISK1.vhdx -SizeBytes 127GB -Dynamic
New-VM -Name "CLIENT-WIN11" -Generation 2 -MemoryStartupBytes 8GB `
    -VHDPath .\CLIENT-WIN11-DISK1.vhdx -SwitchName "LABOPS"

# Monter l'ISO et démarrer dessus
Add-VMDvdDrive -VMName CLIENT-WIN11 -Path "...\Win11_25H2_x64.iso"
Set-VMFirmware -VMName CLIENT-WIN11 -FirstBootDevice (Get-VMDvdDrive -VMName CLIENT-WIN11)

# Exigé par Windows 11 : TPM virtuel + Secure Boot
Set-VMKeyProtector -VMName CLIENT-WIN11 -NewLocalKeyProtector
Enable-VMTPM -VMName CLIENT-WIN11

À retenir

Switch privé → VHDX différentiel sur le parent SYSPREP → New-VM -Generation 2 → mémoire statique. Les serveurs réutilisent l'image ; le client Windows 11 part d'un disque neuf avec TPM + Secure Boot.

03

Réseau & renommage

Après le premier démarrage, chaque serveur reçoit son identité réseau définitive : adresse IP fixe, serveur DNS, désactivation d'IPv6 si inutile, puis renommage et redémarrage.

PowerShell — post-déploiement
# 1. Identifier la carte
Get-NetAdapter

# 2. IP fixe + passerelle
New-NetIPAddress -InterfaceAlias "Ethernet" -IPAddress 172.16.50.10 `
    -PrefixLength 24 -DefaultGateway 172.16.50.1

# 3. Serveur DNS (le DC pointe vers lui-même)
Set-DnsClientServerAddress -InterfaceAlias "Ethernet" -ServerAddresses 172.16.50.10

# 4. Désactiver IPv6 (labo)
Disable-NetAdapterBinding -Name "Ethernet" -ComponentID ms_tcpip6

# 5. Renommer puis redémarrer
Rename-Computer -NewName "SRV-AD01" -Restart

Ordre des opérations

Toujours fixer réseau + nom avant de promouvoir le domaine ou de joindre une machine. Renommer un contrôleur de domaine après promotion est bien plus délicat.

04

Promotion AD en script

La promotion d'un contrôleur de domaine et la création de toute l'arborescence (OU, groupes, utilisateurs) se scriptent intégralement — ce qui rend l'environnement reconstructible à l'identique.

PowerShell — SRV-AD01
Install-WindowsFeature AD-Domain-Services, DNS -IncludeManagementTools
Install-ADDSForest -DomainName "orion.local" -DomainNetbiosName "ORION" -InstallDns

# Arborescence : OU racine → départements → groupes → users
New-ADOrganizationalUnit -Name "ORION-DPT" -Path "DC=orion,DC=local"
New-ADOrganizationalUnit -Name "Management"  -Path "OU=ORION-DPT,DC=orion,DC=local"
New-ADOrganizationalUnit -Name "Development" -Path "OU=ORION-DPT,DC=orion,DC=local"
New-ADOrganizationalUnit -Name "Accounting"  -Path "OU=ORION-DPT,DC=orion,DC=local"

New-ADGroup -Name "MGMT-GRP" -GroupScope Global -GroupCategory Security `
    -Path "OU=Management,OU=ORION-DPT,DC=orion,DC=local"

New-ADUser -Name "Mathis Thomas" -SamAccountName "mth" `
    -UserPrincipalName "mth@orion.local" `
    -Path "OU=Management,OU=ORION-DPT,DC=orion,DC=local" `
    -AccountPassword (Read-Host -AsSecureString "Mot de passe") -Enabled $true
Add-ADGroupMember -Identity "MGMT-GRP" -Members "mth"

À retenir

Ne jamais écrire un mot de passe en clair dans un script : Read-Host -AsSecureString le saisit de façon masquée. Pour rejouer en masse, on boucle sur un Import-Csv d'utilisateurs.

05

Compte de service managé (GMSA)

Un gMSA (Group Managed Service Account) est un compte de service dont Active Directory gère automatiquement le mot de passe — long, complexe, renouvelé tout seul. Idéal pour SQL Server, IIS ou un planificateur de tâches : plus de mot de passe à connaître ni à faire tourner à la main.

Mise en place (vue dans le labo SharePoint/SQL)

  1. 01
    Créer la clé racine KDSUne seule fois par forêt. Elle permet à AD de générer les mots de passe gMSA.
  2. 02
    Préparer un groupeUne OU « GMSA », un groupe GS_GMSA_SQL dont les membres sont les serveurs autorisés (ex. SRV-SQL01).
  3. 03
    Créer le compte gMSAEn autorisant ce groupe à récupérer le mot de passe managé.
  4. 04
    Installer sur le serveur cibleSur SRV-SQL01 : Install-ADServiceAccount, puis on l'affecte au service.
PowerShell — sur le contrôleur de domaine
# 1. Clé racine KDS — UNE SEULE FOIS dans l'environnement
Add-KdsRootKey -EffectiveTime ((Get-Date).AddHours(-10))

# 2. Groupe des serveurs autorisés (membre : SRV-SQL01)
#    créé dans une OU dédiée « GMSA »

# 3. Le compte managé
New-ADServiceAccount GMSA_SQL -DNSHostName gmsa_sql.marvin.lan `
    -PrincipalsAllowedToRetrieveManagedPassword GS_GMSA_SQL

# 4. Sur SRV-SQL01 :
# Install-ADServiceAccount GMSA_SQL

À retenir

Le gMSA résout le cauchemar des mots de passe de comptes de service : AD les gère et les renouvelle seul. Add-KdsRootKey ne s'exécute qu'une fois par forêt ; le décalage -10h évite d'attendre la propagation.
