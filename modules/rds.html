01

Concepts & rôles RDS

RDS (Remote Desktop Services) déporte l'exécution sur le serveur : le poste client n'affiche qu'une image. Le rôle se découpe en plusieurs services de rôle qui collaborent.

RôleSigleFonction
Session HostRDSHExécute réellement les sessions et les applications des utilisateurs
Connection BrokerRDCBRépartit les connexions, gère les collections et la reconnexion aux sessions
Web AccessRDWAPortail web : l'utilisateur voit ses bureaux et RemoteApp publiés dans le navigateur
GatewayRD GatewayPasserelle sécurisée (HTTPS) pour accéder au RDS depuis Internet sans VPN
LicensingRD LicensingGère les licences d'accès client (RDS CAL)

Session-based vs VDI

Session-based (RDSH)

  • Plusieurs utilisateurs partagent un même serveur.
  • Léger, économique, idéal pour applications métier.
  • Le déploiement de ce labo.

VDI (bureaux virtuels)

  • Une VM complète par utilisateur.
  • Isolation maximale, plus gourmand.
  • Pour postes personnalisés / lourds.

À retenir

Le trio de base : RDSH exécute, RDCB répartit, RDWA publie. La Gateway ajoute l'accès sécurisé depuis l'extérieur. En session-based, plusieurs utilisateurs se partagent un serveur — c'est le modèle le plus courant.

03

Déploiement session-based

Le déploiement se pilote depuis le Server Manager du contrôleur de domaine, qui orchestre l'installation des rôles sur les serveurs distants en une seule passe.

  1. 01
    Ajouter les serveurs au poolSur le DC : Manage → Add Servers, sélectionner les 3 VM (RDSH, RDCB, RDWA).
  2. 02
    Lancer l'installation RDSManage → Add Roles and Features → Remote Desktop Services installation → Standard deployment → Session-based.
  3. 03
    Affecter les rôlesDésigner chaque serveur pour son rôle (RDSH = Session Host, RDCB = Connection Broker, RDWA = Web Access), puis Deploy.
  4. 04
    Vérifier la vue d'ensembleDans Remote Desktop Services → Overview, le schéma du déploiement doit afficher les trois rôles au vert.

Astuce

Tous les serveurs doivent être membres du domaine et visibles dans le pool avant de lancer le déploiement. Le Broker (RDCB) est le point d'orchestration : c'est lui qui « connaît » toute la ferme RDS.

04

Collections & RemoteApp

Une collection de sessions regroupe un ou plusieurs Session Hosts et définit qui y a accès. On y publie ensuite soit un bureau complet, soit des RemoteApp (applications isolées qui s'ouvrent comme des fenêtres locales).

  1. 01
    Créer la collectionRemote Desktop Services → Collections → Tasks → Create Session Collection. Nom : Collection1, choisir les Session Hosts, valider les groupes d'utilisateurs autorisés. Décocher l'option de disques de profil utilisateur si non nécessaire, puis Create.
  2. 02
    Publier des RemoteAppDans la collection : Tasks → Publish RemoteApp Programs, sélectionner les applications à exposer (ex. WordPad, une appli métier).
  3. 03
    Tester côté clientLe client ouvre https://rdwa.orion.local/RDWeb, s'authentifie et lance la RemoteApp — elle s'affiche comme une fenêtre locale.

À retenir

Bureau complet ou RemoteApp : la collection est l'unité de publication et de contrôle d'accès. La RemoteApp donne l'illusion d'une application locale alors qu'elle s'exécute sur le RDSH — idéal pour distribuer une seule appli métier sans publier tout le bureau.

05

Certificat du Web Access

Le portail RD Web Access est servi en HTTPS : il lui faut un certificat. En labo, on en génère un auto-signé pour le nom rdwa.orion.local (en production, on le ferait signer par la CA d'entreprise — voir le module Sécurité / PKI).

PowerShell — sur RDWA
# Certificat auto-signé pour le portail Web Access
New-SelfSignedCertificate -DnsName "rdwa.orion.local" `
    -CertStoreLocation "cert:\LocalMachine\My" `
    -NotAfter (Get-Date).AddYears(2) `
    -KeyLength 4096 -KeyExportPolicy Exportable `
    -FriendlyName "RDWA"

# Puis : certlm.msc → Personal → Certificates pour le consulter/exporter

Labo vs production

Un certificat auto-signé déclenche une alerte sur le client (chaîne de confiance non remontée). En production, on déploie un certificat signé par la CA d'entreprise et distribué par GPO — la même logique de PKI que pour Exchange.
