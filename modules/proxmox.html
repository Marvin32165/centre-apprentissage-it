01

Pourquoi virtualiser, et pourquoi Proxmox

Virtualiser, c'est faire tourner plusieurs serveurs logiques sur une seule machine physique. Un hyperviseur découpe les ressources réelles (CPU, RAM, disque, réseau) et les présente à chaque système invité comme s'il avait son propre matériel.

Hyperviseur de type 1 vs type 2

  • Type 1 (bare-metal) — s'installe directement sur le matériel, sans OS hôte en dessous. Performant et stable. Ex. Proxmox VE, VMware ESXi, Microsoft Hyper-V (rôle).
  • Type 2 (hosted) — s'exécute comme une application au-dessus d'un OS classique. Pratique pour du test sur son PC. Ex. VirtualBox, VMware Workstation.

Proxmox VE est un type 1 : c'est une distribution Debian spécialisée, avec un noyau optimisé, l'hyperviseur KVM pour les VM et LXC pour les conteneurs, le tout piloté par une interface web.

VM ou conteneur : deux façons d'isoler

VM (KVM) — virtualisation matérielle

  • Noyau invité complet et indépendant
  • N'importe quel OS : Windows, BSD, autres Linux
  • Isolation forte, mais plus lourde en RAM/disque
  • Démarrage en dizaines de secondes

Conteneur (LXC) — virtualisation système

  • Partage le noyau de l'hôte (Linux seulement)
  • Très léger : quelques Mo de RAM au repos
  • Démarrage quasi instantané
  • Idéal pour des services Linux (web, DNS, DB)

À retenir

Proxmox = Debian + KVM + LXC + ZFS, gratuit et open source. VM quand il faut un OS complet ou non-Linux ; conteneur LXC quand on veut un service Linux léger et rapide.

02

Installation & premier accès web

On grave l'ISO Proxmox VE sur une clé USB, on boote dessus, et l'installeur graphique fait le reste. Deux points comptent : bien choisir le système de fichiers (ZFS si plusieurs disques) et corriger les dépôts après coup.

Étapes de l'installation

  • 01
    Télécharger & graver l'ISORécupère l'ISO sur proxmox.com, écris-la sur une clé USB (Rufus, Ventoy ou dd), puis boote dessus.
  • 02
    Choisir le disque & le filesystemSur un seul disque : ext4 suffit. Sur plusieurs disques : choisis ZFS (RAID1 = miroir, RAIDZ = parité) pour la résilience et les snapshots.
  • 03
    Réseau & identitéRenseigne le hostname (FQDN, ex. pve01.homelab.lan), une IP fixe, la passerelle et le DNS. Le mot de passe root et un email d'admin.
  • 04
    Accéder à l'interfaceAprès reboot, ouvre https://IP-DU-SERVEUR:8006 dans un navigateur. Connexion en root / realm Linux PAM. Le certificat auto-signé déclenche un avertissement : normal en labo.

Corriger les dépôts (sans abonnement)

Par défaut Proxmox pointe vers le dépôt entreprise (payant). En labo, on bascule sur le dépôt no-subscription pour recevoir les mises à jour :

Shell — nœud Proxmox (root)
# Désactiver le dépôt entreprise
sed -i 's/^deb/#deb/' /etc/apt/sources.list.d/pve-enterprise.list

# Ajouter le dépôt no-subscription (Debian 12 « bookworm »)
echo "deb http://download.proxmox.com/debian/pve bookworm pve-no-subscription" \
  > /etc/apt/sources.list.d/pve-no-sub.list

apt update && apt -y full-upgrade

Astuce

Le pop-up « No valid subscription » est purement cosmétique : Proxmox reste 100 % fonctionnel en no-subscription. Un abonnement n'apporte que le dépôt « enterprise » testé et le support.

À retenir

Interface web sur le port 8006 en HTTPS. Après l'install, on désactive le dépôt enterprise et on active no-subscription pour pouvoir mettre à jour.

03

Créer une machine virtuelle (KVM)

Une VM a besoin d'un disque, de RAM, de vCPU, d'une carte réseau et d'un média d'installation. Les bons réglages « virtio » et l'agent invité font toute la différence sur les performances.

L'assistant « Create VM », onglet par onglet

  • General — un VM ID (ex. 100) et un nom.
  • OS — l'ISO uploadée (stockage local → ISO Images). Pour Windows, cocher aussi l'ISO des pilotes virtio-win.
  • System — machine q35, BIOS OVMF (UEFI) pour du récent, contrôleur disque VirtIO SCSI single. Cocher QEMU Agent.
  • Disks — bus SCSI, cache Default (no cache), cocher Discard (TRIM) sur du SSD/ZFS.
  • CPU — type host pour exposer toutes les instructions du processeur physique (perf max en labo).
  • Memory — fixe, ou ballooning pour rendre la RAM inutilisée à l'hôte.
  • Network — modèle VirtIO (paravirtualized), bridge vmbr0.

Créer une VM en ligne de commande

qm — gestion des VM
# Créer la VM 100 : 2 vCPU, 4 Go RAM, réseau virtio sur vmbr0
qm create 100 --name "srv-web" --memory 4096 --cores 2 \
  --net0 virtio,bridge=vmbr0 --scsihw virtio-scsi-single

# Ajouter un disque de 32 Go sur le stockage local-lvm
qm set 100 --scsi0 local-lvm:32,discard=on

# Attacher l'ISO en lecteur CD, puis démarrer
qm set 100 --ide2 local:iso/debian-12.iso,media=cdrom
qm start 100

Cloud-init & templates : déployer en série

Une VM peut devenir un template (clic droit → Convert to template). Combinée à cloud-init (image « cloud » du distributeur), on déploie une nouvelle VM déjà configurée (hostname, IP, clé SSH, utilisateur) en quelques secondes par clonage.

Astuce

Installe qemu-guest-agent dans la VM (apt install qemu-guest-agent). Proxmox affiche alors l'IP de la VM, peut la figer proprement pour un backup cohérent, et l'arrêter/redémarrer « en douceur ».

À retenir

Pour de bonnes perfs : disque SCSI + VirtIO SCSI single, carte réseau VirtIO, CPU host, et l'agent invité installé. Un template + cloud-init transforme le déploiement en clonage express.

04

Les conteneurs LXC

Un conteneur LXC partage le noyau de l'hôte : pas de matériel virtuel à émuler, donc un démarrage instantané et une empreinte minuscule. Parfait pour multiplier les petits services Linux sur un seul serveur.

Créer un conteneur

On télécharge d'abord un template (Datacenter → stockage local → CT Templates → Templates), puis on crée le CT avec ce template.

pct — gestion des conteneurs
# Lister & télécharger un template Debian 12
pveam update
pveam available --section system
pveam download local debian-12-standard_12.7-1_amd64.tar.zst

# Créer le conteneur 200 : 1 Go RAM, 8 Go disque, réseau DHCP
pct create 200 local:vztmpl/debian-12-standard_12.7-1_amd64.tar.zst \
  --hostname ct-dns --memory 1024 --rootfs local-lvm:8 \
  --net0 name=eth0,bridge=vmbr0,ip=dhcp --unprivileged 1

pct start 200
pct enter 200   # ouvre un shell dans le conteneur

Privileged vs unprivileged

  • Unprivileged (recommandé) — le root du conteneur n'est pas le root de l'hôte (mappage d'UID). Bien plus sûr : c'est le défaut.
  • Privileged — root = root hôte. À réserver à des cas précis (montages spéciaux), en toute connaissance de cause.

À retenir

LXC = service Linux léger et instantané, mais lié au noyau de l'hôte (pas de Windows, pas de noyau custom). Garde-les unprivileged par défaut. Pour Docker, préfère une VM plutôt qu'un conteneur LXC.

05

Stockage & ZFS

Proxmox sépare le type de stockage (où et comment sont posées les données) de son usage (disques de VM, disques de conteneurs, ISO, backups, templates). Comprendre cette matrice évite bien des erreurs.

Les principaux types de stockage

TypeDisques VMSnapshotsFichiers (ISO/backup)Usage typique
Directoryqcow2via qcow2OuiSimple, un dossier
LVM-ThinOuiOuiNonDéfaut local-lvm
ZFSOuiOui (instantanés)Via datasetRésilience + snapshots

Ceph RBDOuiOuiNonCluster distribué

ZFS : le combo résilience + snapshots

ZFS est à la fois un système de fichiers et un gestionnaire de volumes RAID logiciel. Ses atouts : sommes de contrôle (détection de corruption silencieuse), snapshots quasi gratuits, compression transparente et self-healing sur un miroir.

ZFS — pool en miroir
# Créer un pool « tank » en miroir sur deux disques
zpool create tank mirror /dev/sdb /dev/sdc

# Activer la compression (gratuit, souvent bénéfique)
zfs set compression=lz4 tank

# État du pool & espace
zpool status tank
zpool list

RAM : ZFS aime la mémoire (l'ARC met en cache les lectures). Compter ~1 Go de RAM par To de données comme ordre de grandeur en homelab.

À retenir

local = fichiers (ISO, backups, templates). local-lvm = disques de VM/CT. Pour la résilience et des snapshots instantanés, monte un pool ZFS en miroir et active la compression lz4.

06

Le réseau : bridges & VLAN

Par défaut, Proxmox crée un pont réseau Linux vmbr0 qui relie la carte physique du serveur. Chaque VM/CT s'y branche comme sur un switch virtuel, et se retrouve sur le même réseau que l'hôte.

Carte physiqueenp1s0Lien vers le switch réel
vmbr0192.168.1.10/24Pont Linux — porte l'IP de l'hôte + relie les VM/CT
VM 100192.168.1.20net0 → vmbr0 (VirtIO)
CT 200192.168.1.21eth0 → vmbr0

Configuration du pont

Tout se règle dans Nœud → System → Network (ou le fichier /etc/network/interfaces). Un pont typique :

/etc/network/interfaces
auto vmbr0
iface vmbr0 inet static
    address  192.168.1.10/24
    gateway  192.168.1.1
    bridge-ports     enp1s0
    bridge-stp       off
    bridge-fd        0
    bridge-vlan-aware yes   # pont conscient des VLAN 802.1Q

Segmenter avec des VLAN

Avec un pont VLAN-aware, on assigne un tag VLAN par carte de VM : dans les options réseau de la VM, champ VLAN Tag = 20 par exemple. Le switch physique doit alors être en mode trunk sur le port du serveur. C'est le lien direct avec le module CCNA (VLAN & 802.1Q).

À retenir

vmbr0 est un switch virtuel : il porte l'IP de l'hôte et relie les VM/CT. Pour segmenter, active bridge-vlan-aware et pose un VLAN Tag par interface — le port physique côté switch doit être en trunk.

07

Sauvegardes & snapshots

Ne jamais confondre les deux : un snapshot est un point de restauration sur place (utile avant une manip risquée) ; une sauvegarde est une copie ailleurs, seule vraie protection contre la perte du serveur.

Snapshots

Sur un stockage qui les supporte (ZFS, LVM-Thin, qcow2), un snapshot fige l'état de la VM en un clic. On peut inclure la RAM (VM en marche) et « rollback » si la mise à jour tourne mal. Ce n'est pas une sauvegarde : si le disque meurt, le snapshot meurt avec.

Sauvegardes vzdump

Le moteur natif vzdump exporte VM et CT en archives. Trois modes :

  • Snapshot — sans interruption (recommandé, nécessite l'agent invité pour la cohérence).
  • Suspend — met en pause le temps de la copie.
  • Stop — éteint la VM (backup le plus « propre », mais coupe le service).
vzdump — sauvegarde manuelle
# Sauvegarder la VM 100, compressée, mode snapshot, vers le stockage backup
vzdump 100 --mode snapshot --compress zstd --storage backup-nas

# Restaurer sous un nouvel ID (110)
qmrestore /mnt/backup/dump/vzdump-qemu-100-*.vma.zst 110

Planifier & Proxmox Backup Server

Dans Datacenter → Backup, on programme des jobs (ex. tous les jours à 2h) avec une politique de rétention. Pour aller plus loin, Proxmox Backup Server (PBS) ajoute la déduplication, les backups incrémentaux et la vérification d'intégrité — idéal quand le volume grossit.

Règle du 3-2-1

3 copies des données, sur 2 supports différents, dont 1 hors site. Un snapshot local ne compte pas comme sauvegarde : teste régulièrement une restauration, pas seulement le backup.

08

Cluster & haute disponibilité

Plusieurs nœuds Proxmox se regroupent en cluster : une seule interface web pilote tout, on migre des VM d'un nœud à l'autre, et avec la HA une VM redémarre automatiquement ailleurs si son nœud tombe.

Créer et rejoindre un cluster

pvecm — cluster manager
# Sur le 1er nœud : créer le cluster
pvecm create homelab-cluster

# Sur chaque nœud suivant : rejoindre via l'IP du 1er nœud
pvecm add 192.168.1.10

# Vérifier l'état & le quorum
pvecm status

Quorum : la règle du nombre impair

Le cluster ne prend de décision que s'il a le quorum (majorité des votes), via Corosync. D'où la règle : un nombre impair de nœuds (3 minimum pour une vraie HA). Avec 2 nœuds, la panne de l'un bloque l'autre — on ajoute alors un QDevice comme 3ᵉ votant.

Migration & haute disponibilité

  • Migration à froid — VM éteinte, déplacée puis redémarrée sur l'autre nœud.
  • Migration à chaud (live) — VM déplacée sans coupure. Nécessite un stockage partagé (NFS, Ceph…) accessible par les deux nœuds.
  • HA — on déclare la VM en ressource HA ; si son nœud meurt, le cluster la relance ailleurs automatiquement.

À retenir

Cluster = gestion unifiée + migration. La HA suppose un stockage partagé et un nombre impair de nœuds (≥ 3) pour garder le quorum. À 2 nœuds, ajoute un QDevice témoin.

09

Exploitation au quotidien

Une fois l'infra en place, l'admin gère les accès, les mises à jour, la surveillance et l'automatisation. Quelques réflexes rendent Proxmox propre et sûr sur la durée.

Utilisateurs, rôles & jetons API

Proxmox a son propre modèle de permissions (Datacenter → Permissions). On crée des utilisateurs dans un realm (PVE ou PAM), on leur affecte des rôles sur des chemins (ex. PVEVMUser sur un pool de VM). Pour l'automatisation (Terraform, Ansible, scripts), on génère un API Token plutôt que d'utiliser le mot de passe root.

Mises à jour & santé

  • apt update && apt full-upgrade régulièrement ; reboot planifié si le noyau change.
  • Surveiller la RAM, la charge et l'espace ZFS ; alerter par email (déjà configuré à l'install).
  • Sur ZFS, lancer un zpool scrub périodique pour détecter/réparer la corruption.

Helper scripts de la communauté

La communauté (ex. projet Proxmox VE Helper-Scripts) fournit des scripts qui déploient un service en LXC en une commande. Pratique, mais à n'exécuter qu'après avoir lu le script : ne colle jamais un curl … | bash à l'aveugle sur ton hyperviseur.

Pour aller plus loin

Deux chaînes de référence pour ce module : Virtualization HowTo (très orientée Proxmox) et TechnoTim (homelab & automatisation). Le module Homelab & Self-hosting prolonge celui-ci côté services.

À retenir

Automatise avec des API tokens (jamais root), maintiens à jour, surveille l'espace ZFS et lance des scrubs. Et relis toujours un script communautaire avant de l'exécuter sur l'hyperviseur.
