01

Rôles & fonctionnalités

Windows Server est une base neutre que l'on spécialise en lui ajoutant des rôles (ce que le serveur fait : contrôleur de domaine, serveur web, DHCP…) et des fonctionnalités (briques de support : outils d'administration, sauvegarde…).

Desktop Experience vs Server Core

Desktop Experience

  • Interface graphique complète (bureau, explorateur).
  • Plus simple à apprendre et à administrer localement.
  • Surface d'attaque et empreinte disque plus grandes.

Server Core

  • Pas d'interface : administration en ligne de commande / à distance.
  • Moins de mises à jour, moins de surface d'attaque.
  • Console de configuration : sconfig.cmd.

Ajouter un rôle — deux voies

Via le Server Manager : Manage → Add Roles and Features, puis on choisit le type d'installation (basé rôle/fonctionnalité ou services Bureau à distance), le serveur cible, et on coche le rôle voulu.

PowerShell — équivalent en script
# Lister les rôles/fonctionnalités disponibles
Get-WindowsFeature

# Installer un rôle + ses outils d'administration
Install-WindowsFeature AD-Domain-Services, DNS -IncludeManagementTools

# Sur Server Core : ouvrir la console de configuration
sconfig.cmd

Astuce labo

Sur le terrain, ne cochez pas « redémarrer automatiquement » pendant l'ajout d'un rôle : vous gardez la main sur le moment du reboot. En PowerShell, ajoutez -Restart seulement quand vous êtes prêt.

02

SYSPREP & déploiement par image

Réinstaller Windows manuellement sur 10 serveurs est une perte de temps. La bonne méthode : préparer une seule image de référence, la « généraliser » avec SYSPREP, puis créer toutes les VM à partir de ce disque parent en quelques secondes via des disques différentiels.

Pourquoi généraliser ?

Chaque installation Windows porte un identifiant unique (SID) et un nom machine. Cloner brutalement un disque dupliquerait ces identifiants — interdit dans un domaine. SYSPREP en mode generalize efface ces éléments propres à la machine : au prochain démarrage, Windows repasse par l'écran de configuration (OOBE) et reçoit une nouvelle identité.

Préparer l'image de référence (exercice du labo)

  1. 01
    Créer la VM modèleVM Hyper-V « SYSPREP » : 8 Go RAM, 2 vCPU, ISO Windows Server 2025, installation Standard (Desktop Experience).
  2. 02
    Pré-configurer WindowsFuseau horaire, désactivation de IE Enhanced Security, IP, Bureau à distance, pare-feu. Partitionner le disque en deux : C:\ ≈ 40 % du VHDX (127 Go) et D:\.
  3. 03
    Lancer SYSPREPExécuter C:\Windows\System32\Sysprep\sysprep.exe en mode OOBE, cocher Généraliser, action Arrêter le système.
  4. 04
    Archiver le disque parentSupprimer la VM d'Hyper-V (on garde le disque), renommer le VHDX en TEST-SYSPREP.vhdx et le déposer dans C:\PARENT.
  5. 05
    Cloner à volontéCréer les VM (ex. LAB-WS01, LAB-WS02) à partir de disques différentiels pointant vers ce parent. Les deux machines doivent pouvoir se ping.
PowerShell — clonage par disque différentiel
# Un disque enfant qui ne stocke QUE les différences avec le parent
New-VHD -Path .\LAB-WS01-DIFF.vhdx `
    -ParentPath "C:\PARENT\TEST-SYSPREP.vhdx" -Differencing

New-VM -Name "LAB-WS01" -Generation 2 -MemoryStartupBytes 8GB `
    -VHDPath .\LAB-WS01-DIFF.vhdx -SwitchName "LABOPS"

À retenir

Image de référence → SYSPREP /generalize /oobe → disque parent figé → disques différentiels. On déploie en secondes, sans dupliquer le SID, et on économise un disque entier par machine.

03

Active Directory Domain Services

AD DS est l'annuaire central du réseau : il authentifie les utilisateurs, organise les ordinateurs et applique des stratégies. Tout part d'une forêt contenant un domaine, structuré en unités d'organisation (OU).

La hiérarchie

  • Forêt — frontière de sécurité la plus haute ; contient un ou plusieurs domaines.
  • Domaine — ex. orion.local (nom NetBIOS ORION) : la base d'authentification commune.
  • Unité d'organisation (OU) — conteneur logique (par département, par site) sur lequel on applique des GPO et on délègue l'administration.
  • Objets — utilisateurs, groupes, ordinateurs.

Promouvoir le premier contrôleur de domaine

PowerShell — SRV-AD01
# Rôle AD DS + DNS
Install-WindowsFeature AD-Domain-Services, DNS -IncludeManagementTools

# Créer la forêt + le domaine + le DNS intégré
Install-ADDSForest -DomainName "orion.local" `
    -DomainNetbiosName "ORION" -InstallDns

Structurer : OU, groupes, utilisateurs

Modèle du labo Orion : une OU racine ORION-DPT, trois départements (Management, Development, Accounting), un groupe de sécurité par département et des utilisateurs rattachés.

PowerShell — arborescence AD
# OU racine + une OU par département
New-ADOrganizationalUnit -Name "ORION-DPT" -Path "DC=orion,DC=local"
New-ADOrganizationalUnit -Name "Development" -Path "OU=ORION-DPT,DC=orion,DC=local"

# Groupe de sécurité global
New-ADGroup -Name "DEV-GRP" -GroupScope Global -GroupCategory Security `
    -Path "OU=Development,OU=ORION-DPT,DC=orion,DC=local"

# Utilisateur (mot de passe saisi de façon masquée)
New-ADUser -Name "Sébastien Sotiaux" -SamAccountName "sso" `
    -UserPrincipalName "sso@orion.local" `
    -Path "OU=Development,OU=ORION-DPT,DC=orion,DC=local" `
    -AccountPassword (Read-Host -AsSecureString "Mot de passe") -Enabled $true

Add-ADGroupMember -Identity "DEV-GRP" -Members "sso"

Joindre une machine au domaine

PowerShell — membre du domaine
Add-Computer -DomainName "orion.local" -Restart
# Alternative en cmd :
# netdom join SRV-FILE01 /domain:orion.local /userd:ORION\Administrator /passwordd:*

À retenir

Forêt → domaine → OU → objets. On applique les GPO et on délègue au niveau de l'OU, on autorise via les groupes. Install-ADDSForest crée tout le socle ; pour le 2e DC on utilise Install-ADDSDomainController.

04

DNS & DHCP

Deux services réseau indissociables d'Active Directory : DNS traduit les noms en adresses IP (et localise les contrôleurs de domaine), DHCP distribue automatiquement les adresses IP aux clients.

DNS — l'annuaire des noms

  • Zone de recherche directe — nom → IP (srv-ad01.orion.local → 172.16.50.10).
  • Zone de recherche inversée — IP → nom.
  • Enregistrements clés : A (hôte), CNAME (alias), MX (messagerie), SRV (services — c'est ainsi que les clients trouvent le DC).
  • AD dépend du DNS : un client ne peut ouvrir une session que s'il résout les enregistrements SRV du domaine.

Configuration réseau d'un serveur

PowerShell — IP fixe + DNS
Get-NetAdapter
New-NetIPAddress -InterfaceAlias "Ethernet" -IPAddress 172.16.50.10 `
    -PrefixLength 24 -DefaultGateway 172.16.50.1
Set-DnsClientServerAddress -InterfaceAlias "Ethernet" -ServerAddresses 172.16.50.10
# Désactiver IPv6 si non utilisé dans le labo
Disable-NetAdapterBinding -Name "Ethernet" -ComponentID ms_tcpip6

DHCP — l'attribution automatique

Le client diffuse une demande (DORA : Discover → Offer → Request → Acknowledge) ; le serveur DHCP lui prête une adresse d'une étendue (scope) pour une durée de bail. En environnement AD, le serveur DHCP doit être autorisé dans l'annuaire avant de distribuer.

À retenir

Sur un contrôleur de domaine, le serveur pointe son DNS vers lui-même. Sans DNS fonctionnel, pas d'ouverture de session AD. DHCP simplifie l'adressage des clients via les 4 étapes DORA.

05

DFS Namespace

DFS (Distributed File System) offre aux utilisateurs un seul chemin réseau logique qui masque l'emplacement physique réel des partages — même s'ils sont répartis sur plusieurs serveurs.

Le problème résolu

Sans DFS, l'utilisateur doit connaître \\SRV-FILE01\Compta, \\SRV-FILE02\RH… Si on déplace un partage, tous les raccourcis cassent. Avec DFS, tout passe par un chemin unique stable : \\orion.local\Partages\Compta.

Namespace (espace de noms)

  • Le chemin logique unique présenté aux utilisateurs.
  • Des dossiers pointent (targets) vers les vrais partages.
  • On peut déplacer le partage réel sans changer le chemin.

Replication (DFS-R)

  • Réplique le contenu entre plusieurs serveurs.
  • Tolérance de panne : si un serveur tombe, l'autre répond.
  • Répartition de charge entre sites.

À retenir

DFS Namespace = abstraction du chemin (un seul nom pour l'utilisateur). DFS Replication = copies synchronisées pour la disponibilité. Les deux se combinent pour un partage de fichiers résilient.

06

Sécurité du serveur

Sécuriser un serveur, c'est réduire la surface d'attaque et appliquer le moindre privilège : n'installer que le nécessaire, ne donner que les droits indispensables, et tout journaliser.

Les bons réflexes

  • Moindre privilège — un compte de service n'a que les droits dont il a besoin (voir les GMSA dans le module PowerShell).
  • Server Core quand l'interface n'est pas requise : moins de composants = moins de failles.
  • Pare-feu Windows Defender — n'ouvrir que les ports des services réellement exposés.
  • Mises à jour — planifier WSUS / Windows Update pour combler les vulnérabilités.
  • GPO de sécurité — stratégies de mots de passe, verrouillage de compte, droits utilisateur.
  • Sauvegarde testée — règle 3-2-1 : 3 copies, 2 supports, 1 hors site.

Le principe de la chaîne de confiance

Beaucoup de sécurité Windows repose sur les certificats et une autorité de certification (CA) interne. Ce sujet est traité en profondeur dans le module Exchange & Sécurité (PKI, chaîne de confiance, AD CS).

Point de vigilance

La sécurité n'est pas une case à cocher : c'est moindre privilège + surface réduite + journalisation + sauvegarde testée, en continu. Une sauvegarde jamais restaurée n'est qu'une hypothèse.
